## Findings

### [CRITICAL] Null dereference in Get-ClassificationConfig crashes pipeline under StrictMode
- **File:** `scripts/06-ClassifyThemes.ps1:65,84,98`
- **Issue:** `Get-ClassificationConfig` returns `$null` when `classification` section is absent from config (line 65). Caller assigns to `$classCfg` (line 84) then accesses `$classCfg.Enabled` (line 98). Under `Set-StrictMode -Version Latest`, property access on `$null` throws `PropertyNotFoundException`, crashing the stage. The existing unit test `Test-ArchiveConfigSchema (valid)` constructs a config with classification `enabled = $false`, so the section exists and the null path is never exercised.
- **Fix:** Change line 65 to return a default `[pscustomobject]` with `Enabled = $false` instead of `$null`, matching the pattern used by `New-ArchiveRun` for missing config sections:

```powershell
if (-not $Config.classification) {
    return [pscustomobject]@{
        Enabled = $false
        Model = "qwen3:4b"
        MaxChars = 6000
        Endpoint = "http://localhost:11434"
        SystemPrompt = "You are an archive analyst..."
    }
}
```

### [CRITICAL] Path traversal via `destination` column in approved manifest
- **File:** `scripts/09-ApplyReviewedActions.ps1:51-53`
- **Issue:** When `action` is `quarantine` or `move` and the CSV row has a non-empty `destination` field, the script uses it directly as the destination root at line 52: `$destinationRoot = if (-not [string]::IsNullOrWhiteSpace($row.destination)) { $row.destination } else { $quarantinePath }`. A crafted `destination` value like `../../etc/` could move files outside the quarantine area. The approved manifest CSV is human-reviewed but the path is not validated against an allowlist.
- **Fix:** Validate `$row.destination` is a subdirectory of an allowed quarantine base path, or reject user-supplied destinations and always use the configured quarantine path:

```powershell
$destinationRoot = $quarantinePath  # ignore CSV destination, use config
# OR
if (-not [string]::IsNullOrWhiteSpace($row.destination)) {
    $resolvedDest = Resolve-ArchivePath -PathValue $row.destination -BasePath $run.ToolkitRoot
    $quarantineBase = Resolve-ArchivePath -PathValue $quarantinePath -BasePath $run.ToolkitRoot
    if (-not (Test-PathInside -ChildPath $resolvedDest -ParentPath $quarantineBase)) {
        throw "Destination path must be inside quarantine directory"
    }
    $destinationRoot = $resolvedDest
}
```

### [HIGH] TOCTOU race between Test-Path and Move-Item in ApplyReviewedActions
- **File:** `scripts/09-ApplyReviewedActions.ps1:46,62`
- **Issue:** `Test-Path` at line 46 confirms the file exists. Between that check and `Move-Item` at line 62, an attacker could replace the file with a symlink to an arbitrary target, causing the move operation to follow the symlink. This is a classical time-of-check-to-time-of-use vulnerability.
- **Fix:** Use `Move-Item` directly in a try/catch without pre-checking existence, or use a transactional approach. The `-ErrorAction Stop` on Move-Item handles failure, so the pre-check is redundant:

```powershell
# Remove the Test-Path check (line 46-49) and just attempt the move
try {
    Move-Item -LiteralPath $row.path -Destination $destination -ErrorAction Stop
    ...
}
catch [System.IO.FileNotFoundException] {
    $results.Add(... status = "missing" ...)
}
```

### [HIGH] All module functions publicly exported, including internal helpers
- **File:** `scripts/common/ArchiveAgent.Common.psm1:925-933`
- **Issue:** `Export-ModuleMember -Function *` at line 925 exports every function in the module. The subsequent targeted exports at lines 926-933 are redundant because `-Function *` already covers them. Internal functions like `Compress-ArchivePerceptualHash`, `Get-ArchivePerceptualHash`, `Compare-ArchivePerceptualHash`, `ConvertTo-ArchiveMarkdownValue`, `Test-PathInside`, `Test-ArchiveExcluded`, `Get-ArchiveFiles`, `Get-ArchiveHash`, `Start-ArchiveTimer`, and `Stop-ArchiveTimer` become part of the public API surface. This pollutes the global namespace and creates unversioned API commitments.
- **Fix:** Remove `Export-ModuleMember -Function *` and list only the intended public API functions:

```powershell
Export-ModuleMember -Function @(

    'Start-ArchiveTimer', 'Stop-ArchiveTimer', 'Test-ArchiveConfigSchema',
    'ConvertFrom-ArchiveLlmJson', 'Invoke-ArchiveLlm',
    'New-ArchiveRun', 'Write-ArchiveLog',
    'Get-ArchiveToolkitRoot', 'Resolve-ArchivePath', 'Read-ArchiveConfig',
    'Test-ArchiveCsvColumns', 'Import-ArchiveCsv', 'Export-ArchiveCsv',
    'Ensure-ArchiveDirectory', 'Get-ArchiveFileCategory', 'Get-ArchiveFiles',
    'Get-ArchiveHash', 'Get-ArchiveSafeStem', 'Test-ArchiveExcluded',
    'Test-ArchiveCommand', 'Get-ArchiveToolVersion',
    'Invoke-ArchiveConfiguredCommand', 'ConvertTo-ArchiveMarkdownValue',
    'Get-ArchivePerceptualHash', 'Compress-ArchivePerceptualHash', 'Compare-ArchivePerceptualHash',
    'Test-ArchiveSystemPath', 'Test-PathInside'
)
```

### [MEDIUM] External tool stderr leaked to CSV error logs
- **File:** `scripts/common/ArchiveAgent.Common.psm1:756` (and used in `scripts/02-Metadata.ps1:167`, `scripts/04-ExtractText.ps1:93`, `scripts/05-TranscribeMedia.ps1:56`)
- **Issue:** `Invoke-ArchiveConfiguredCommand` returns raw combined stdout+stderr from external tools (line 756: `Output = ($output -join [Environment]::NewLine)`). Callers like `02-Metadata` at line 167 write this to CSV error logs: `error = $result.Output + $result.Error`. If ffprobe, exiftool, or tesseract emit file paths, metadata values, or internal error messages to stderr, those are persisted in CSV files under `outputs/`. For a local archive tool this is low-severity, but if outputs are shared, this leaks file paths and system state.
- **Fix:** Truncate or sanitize error output before writing to CSV logs. Or separate stdout (tool output) from stderr (diagnostics) using different streams:

```powershell
# In Invoke-ArchiveConfiguredCommand, separate stdout and stderr
$output = & $command @args 2>$stderrTempFile
$errorText = Get-Content -LiteralPath $stderrTempFile -Raw
return [pscustomobject]@{
    Available = $true
    ExitCode = $exitCode
    Output = ($output -join [Environment]::NewLine)
    Error = $errorText
}
```

### [MEDIUM] Config validation skips range check when similarityThreshold is 0.0
- **File:** `scripts/common/ArchiveAgent.Common.psm1:167-171`
- **Issue:** The validation condition `if ($Config.dedupe.PSObject.Properties.Name -contains "similarityThreshold" -and $Config.dedupe.similarityThreshold)` uses `-and $Config.dedupe.similarityThreshold` as a truthiness check. When `similarityThreshold` is `0.0` (a valid value within the allowed range 0-1), the condition evaluates to `$false`, skipping the range validation. A threshold of 0.0 silently passes without verifying it's within bounds.
- **Fix:** Remove the truthiness check:

```powershell
if ($Config.dedupe.PSObject.Properties.Name -contains "similarityThreshold") {
    $threshold = $Config.dedupe.similarityThreshold
    if ($null -ne $threshold -and ($threshold -lt 0 -or $threshold -gt 1)) {
        $errors += "dedupe.similarityThreshold must be between 0 and 1"
    }
}
```

### [MEDIUM] ChromaDB collection silently dropped on re-index
- **File:** `scripts/python/11_semantic_search.py:527-532`
- **Issue:** Index mode deletes any existing ChromaDB collection with the same name (`client.delete_collection(name)`) and creates a fresh one. This happens silently — the only log message is "Storing in ChromaDB: <path>" at INFO level (line 662). Users who accidentally run index mode will lose their existing search index without clear warning. The broad `except ValueError` at line 531 also swallows unrelated errors.
- **Fix:** Add an explicit warning before deletion, narrow the exception handler, and consider requiring confirmation in interactive mode:

```python
try:
    existing = client.get_collection(name)
    log.warning("Dropping existing collection '%s' with %d document(s). Re-run to re-index.", name, existing.count())
except ValueError:
    pass  # collection does not exist
```

### [LOW] Unnecessary CSV reads during DryRun in Cleanup stage
- **File:** `scripts/10-Cleanup.ps1:55-66,70-82`
- **Issue:** The stage reads extraction-status.csv and transcription-status.csv before checking the `$DryRun` flag. While harmless for correctness, this performs disk I/O that is discarded during a dry run. If the CSVs are large (thousands of files), this adds latency for no benefit.
- **Fix:** Move the CSV reading inside `if (-not $DryRun)` blocks, or restructure so dry-run skips CSV loading entirely.

### [LOW] Czkawka output parsing uses fragile regex matching
- **File:** `scripts/03-Dedupe.ps1:163-165`
- **Issue:** Czkawka output is parsed with `$_ -match '\.(jpg|jpeg|png|gif|webp|heic|tif|tiff|bmp|mp4|mov|mkv)'`. This matches any line containing a file extension string, not specifically Czkawka similarity lines. Lines from progress output, errors, or tool headers could produce false-positive near-duplicate entries. The regex `-->` delimiter is literal and fine, but the extension match is too permissive.
- **Fix:** Match against a more specific Czkawka output format or validate the line structure before extracting paths:

```powershell
$czkawkaLines = $czkawkaText -split [Environment]::NewLine | Where-Object {
    $_ -match '^\[.*\]\s+.+\.(jpg|jpeg|png)\s+-->\s+.+\.(jpg|jpeg|png)$'
}
```

### [LOW] Model download on every entity extraction run
- **File:** `scripts/python/10_extract_entities.py:400`
- **Issue:** `_GLiNER.from_pretrained(args.model)` relies on HuggingFace's implicit local cache (`~/.cache/huggingface/`). If the cache is cleared or the user is offline, the script fails with a generic `requests.exceptions.ConnectionError`. No explicit cache check or offline-mode fallback is provided.
- **Fix:** Check for cached model path before downloading, and provide a clear error message pointing to the cache location:

```python
import os
from huggingface_hub import try_to_load_from_cache
cached = try_to_load_from_cache(args.model, "pytorch_model.bin")
if cached is None or not os.path.isfile(cached):
    log.warning("Model '%s' not found in local cache. Downloading...", args.model)
```

## Summary
- **2 critical, 2 high, 3 medium, 3 low** findings
- **Overall assessment:** The codebase is well-structured with strong safety mechanisms (StrictMode, output isolation, explicit opt-in for destructive operations). The critical issues are both in the PowerShell pipeline: a null dereference that will crash stage 06 when classification is disabled (the default), and a path traversal vector in stage 09's CSV destination field. The high-severity issues (TOCTOU race, over-exported module API) are architectural hardening concerns. Most medium/low findings relate to error handling, data leakage, and parsing fragility. The Python sidecars are well-written with proper lazy imports and structure.
